# Roadmap: v5.0 Hg1223 Instrumented Reproduction and Basin Test

## Overview

Milestone `v4.0` closed with `Hg1223` still primary, bilayer nickelates still backup, and a concrete next-step memo: run the first instrumented low-`TQ` benchmark-window campaign rather than inventing another route ranking. This milestone turns that memo into an experiment-facing package with explicit run ordering, logging, failure localization, and route gates. It closes without a room-temperature discovery, but with a much tighter first-campaign decision framework.

## Phase Details

### Phase 19: Hg1223 Instrumented Stage-A Runbook and Logging Schema

**Goal:** Convert the carried Stage `A` benchmark-window campaign into a collaborator-facing runbook with exact nodes, ordered steps, mandatory logs, and run invalidation logic.
**Depends on:** Phase `17` and Phase `18` outputs
**Requirements:** `RUN-01`, `RUN-02`, `RUN-03`, `VALD-01`, `VALD-02`
**Contract Coverage:**
- Advances: `claim-hg1223-runbook`
- Deliverables: `deliv-phase19-runbook`, `deliv-phase19-log-schema`
- Acceptance tests: `test-runbook-explicit`
- Anchor coverage: `ref-phase17-sweep`, `ref-phase17-sequence`, `ref-phase17-gates`, `ref-phase18-next-step`, `ref-hg1223-quench`
- Forbidden proxies: `fp-run-without-vq`

**Success Criteria**

1. The runbook states the exact carried Stage `A` nodes and their run order
2. `PQ`, `TQ`, `vQ`, sample metadata, and thermal-path logging are mandatory rather than optional
3. Cryogenic-first handling classes and run invalidation rules are explicit
4. The outputs are collaborator-facing rather than internal shorthand

**Backtracking trigger:** If the runbook still depends on hidden operator skill or leaves `vQ` qualitative, Phase `20` must tighten the diagnostic package and lower confidence in near-term reproducibility.

**Plans:** 3/3 plans complete

**Executed outcome (2026-03-29):** Phase `19` now gives the repo a collaborator-facing Stage `A` package instead of an internal sketch. The carried `Hg1223` benchmark-window nodes are fixed into one runbook, the logging schema makes `PQ`, `TQ`, `vQ`, sample metadata, and thermal-path history mandatory, and the handling spec now separates invalidated runs from partial-but-interpretable runs. The next work is no longer "what should we run?" but "how do we localize failure once these runs are instrumented?"

### Phase 20: Hg1223 Failure-Mode Diagnostics and Minimum Evidence Package

**Goal:** Define how the first instrumented campaign should distinguish target-state failure, quench-trajectory failure, sample-state failure, retrieval loss, and onset-only overclaiming.
**Depends on:** Phase `19`
**Requirements:** `DIAG-01`, `DIAG-02`, `DIAG-03`, `VALD-01`, `VALD-02`
**Contract Coverage:**
- Advances: `claim-hg1223-diagnostics`
- Deliverables: `deliv-phase20-diagnostics`, `deliv-phase20-evidence-package`
- Acceptance tests: `test-diagnostics-localize-failure`
- Anchor coverage: `ref-phase17-sequence`, `ref-phase17-gates`, `ref-hg1223-quench`, Phase `19` outputs
- Forbidden proxies: `fp-diagnostics-without-stage-localization`

**Success Criteria**

1. Each main failure mode has stage-local diagnostic signatures
2. The minimum evidence package distinguishes interpretable runs from noise
3. The repo states which measurements are needed for a first retained-state claim and which are second-stage upgrades
4. Onset-only and poorly logged runs are explicitly rejected as decisive evidence

**Backtracking trigger:** If the diagnostic package still cannot separate route failure from handling failure, Phase `21` must strengthen pivot logic and lower route confidence.

**Plans:** 3/3 plans complete

**Executed outcome (2026-03-29):** Phase `20` now separates invalid runs, route-relevant misses, fragile retained hits, and basin-supporting evidence instead of letting everything collapse into generic "reproducibility" language. The repo now has a stage-local failure map, a `T0-T3` evidence ladder, and a diagnostic routing tree that says what each observed outcome does and does not justify.

### Phase 21: First-Campaign Route Gates and Backup Trigger Memo

**Goal:** Convert the runbook and diagnostic package into an honest post-campaign route memo that preserves the `149 K` room-temperature gap and the nickelate backup.
**Depends on:** Phase `19` and Phase `20`
**Requirements:** `GATE-01`, `GATE-02`, `GATE-03`, `VALD-03`
**Contract Coverage:**
- Advances: `claim-route-pivot-criteria`
- Deliverables: `deliv-phase21-route-gates`, `deliv-phase21-final-memo`
- Acceptance tests: `test-route-gates-honest`
- Anchor coverage: `ref-phase14-decision`, `ref-phase17-gates`, `ref-lapr327-60k`, Phase `19` and `20` outputs
- Forbidden proxies: `fp-route-pivot-without-gap`

**Success Criteria**

1. The route memo states when `Hg1223` stays primary, when it is held without upgrade, and when it is downgraded
2. The nickelate backup remains explicit and current
3. The `149 K` room-temperature gap remains explicit
4. The milestone ends with a concrete next action rather than vague optimism

**Backtracking trigger:** If the route gates can only be written by hiding missing controls or the room-temperature gap, the milestone should not close.

**Plans:** 3/3 plans complete

**Executed outcome (2026-03-29):** Phase `21` closes `v5.0` with explicit keep-primary, hold-confidence, strengthened-route, and downgrade actions for `Hg1223`, all tied to the Phase `20` evidence ladder. The bilayer nickelate backup is preserved with a concrete activation trigger, and the `149 K` room-temperature gap remains explicit in the closeout logic.
